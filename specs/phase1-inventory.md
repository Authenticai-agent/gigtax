# Phase 1 inventory — carry-over and scaffold audit

Status: **complete**. `npm run build` produces 53 static pages; `astro check` reports 0 errors.

## What `legacy/` contained

- **`legacy/js/tax-engine.js`** — the calculation core: 24 pure functions (federal
  income tax, SE tax, FICA, QBI, state tax, SSDI, ACA, capital gains, entity
  comparison, S-corp optimizer, quarterly estimated, 1099-K reconciliation,
  Delaware franchise tax, etc.). No dependencies beyond the config object.
- **`legacy/js/app.js`** — ~9,200 lines of vanilla JS: ~97 view functions and a
  hash/path router with 483 route entries, building HTML as strings. **No React,
  no framework, no build step.**
- **`legacy/data/config.json`** (156 KB) — the authoritative runtime dataset, fetched
  at runtime by the old engine. Top-level keys: `federal`, `retirement`,
  `selfEmploymentDeductions`, `entityTypes`, `incomeSources`, `acaSubsidy`,
  `platforms`, `states`, `filingStatuses`, `professionals`, `equityCompensation`,
  `combinedSalaryCalculator`, `stateMetadata`, and others. `_meta` carries a
  correction audit trail through **v3.3 (last verified 2026-06-10)**.
- **`legacy/gigtax-2026-config.json`** — a near-duplicate of `data/config.json` that
  differs in content. Treated as stale; `data/config.json` is the source of truth.
- **`legacy/gigtax-build-plan-v4-complete.html`** — design/spec document.
- CSS design tokens in `legacy/css/styles.css`; one-off migration scripts
  (`fix-*.js/py`, `update-standalone-*.js`) with hard-coded macOS paths — historical.

## Needs owner review / decisions

1. **No React in `legacy/` — approach taken.** The spec/CLAUDE.md assume "React
   calculator islands" and Phase 1 says to *copy* calculator components. None exist.
   Resolution: the calculation **logic** (`tax-engine.js`) is the asset worth
   preserving and was ported verbatim; the calculator **UI** is necessarily new
   React (`QuickEstimator`). Confirm this is the intended reading before Phase 2
   builds more islands.
2. **Platform worked-example incomes are missing from the data.** Legacy platform
   entries have `incomeType` and `keyDeductions` but **no example income** and no
   display names. Per ground rule #2, Tier-3 platform pages (Phase 5) are blocked
   until the owner supplies example incomes (and ideally richer deduction
   descriptions). `slug`/`name` were derived and can be overridden.
3. **Calculation bug fixes (Phase 2).** Owner approved fixing bugs; done using only
   figures already in the verified dataset, verified numerically:
   - ✅ **Fixed:** flat-state exempt-below threshold now honored (Ohio exempts income
     at/below `$26,050`, taxing only the excess at 2.75%). Only Ohio has this field;
     other flat states are unaffected. Ohio state tax on a $50k example dropped from
     $1,278 → $561.
   - ✅ **Fixed:** 0.9% Additional Medicare Tax (Form 8959) now applied in `calcSETax`
     and `calcFICA` above the status threshold ($200k single / $250k MFJ / $125k MFS,
     from the dataset). The deductible half of SE tax is still computed on the regular
     portion only. Below threshold, results are identical to before (zero blast radius
     for typical incomes).
   - ⏳ **Deferred (needs verified data):** `calcEIC` MFJ phase-out. The dataset has no
     MFJ EIC figures — only a prose "~$7,270 higher" note — so a correct fix needs the
     2026 MFJ phase-out-start amounts (4 values) from IRS Rev. Proc. 2025-32 added to
     the dataset. Not invented per ground rule #2. No live calculator uses EIC yet.
4. **Scale vs. quality tension.** The state pages prove the SSG pattern that reaches
   1,000+ pages. But the requested `/working-in-one-state-living-in-another/:state-pair`
   layer is 50×51 = 2,550 near-duplicate pages, which the spec's own constraint #5
   forbids ("more than two axes… risks a quality classification"). Needs a decision
   before Phase 3/4 scale-out.
